## 3. 开发规范

### 3.1 代码风格

- **命名规范**：
  - 类名：使用大驼峰命名法（如 `PikafishAI`）
  - 方法名：使用小驼峰命名法（如 `initialize()`）
  - 变量名：使用小驼峰命名法（如 `selectedPieceID`）
  - 常量名：使用全大写加下划线（如 `MIN_CLICK_DELAY_TIME`）

- **代码格式**：
  - 缩进：使用4个空格
  - 行宽：建议不超过120字符
  - 空行：在方法之间、逻辑块之间添加空行
  - 注释：关键代码添加注释说明

### 3.2 日志规范

- **日志工具**：使用 `Utils.LogUtils` 类
- **日志标签**：使用有意义的标签，如 `PikafishAI`、`PvMActivity`
- **日志级别**：
  - `v()`：详细调试信息
  - `d()`：调试信息
  - `i()`：一般信息
  - `w()`：警告信息
  - `e()`：错误信息

- **日志内容**：
  - 包含时间戳、级别、标签、消息
  - 关键操作和错误必须记录
  - 避免过度日志，影响性能

### 3.3 错误处理

- **异常处理**：使用 try-catch 捕获异常
- **错误日志**：异常必须记录到日志
- **用户反馈**：重要错误需要向用户显示Toast提示
- **恢复机制**：关键操作失败后应有恢复机制

### 3.4 性能优化

- **线程管理**：耗时操作（如AI计算）放在后台线程
- **内存管理**：避免内存泄漏，及时释放资源
- **UI优化**：避免主线程阻塞，使用异步操作
- **资源使用**：合理使用系统资源，避免过度消耗

## 5. AI集成规范

- **二进制文件**：使用 `app/src/main/assets/` 中的二进制文件
- **初始化流程**：
  1. 复制二进制文件到应用目录
  2. 设置可执行权限
  3. 启动进程
  4. 发送UCI命令初始化
  5. 等待初始化完成

  - 生成FEN字符串表示棋盘状态
  - 发送位置信息和思考命令

  - 验证走法合法性

- **初始化失败**：记录详细日志，提供备用方案

- **日志分析**：使用Logcat分析应用运行状态
- **文件检查**：检查应用生成的日志文件

### 6.3 日志收集操作步骤

1. **创建logs目录**：
   ```bash
   mkdir -p /Users/yy/pro-test/ChineseChess/logs
   ```

2. **开启ADB日志监控**：
   ```bash
   adb logcat -s 'PvMActivity:*' 'PikafishAI:*' 'LogUtils:*' '*:E'
   ```

3. **将日志保存到指定文件**：
   ```bash
   adb logcat -s 'PvMActivity:*' 'PikafishAI:*' 'LogUtils:*' '*:E' > /Users/yy/pro-test/ChineseChess/logs/test_log_$(date +%Y%m%d_%H%M%S).txt
   ```

4. **执行测试操作**：
   - 在应用中执行测试步骤中的操作
   - 操作过程中，ADB会实时记录日志

5. **停止日志收集**：
   - 当测试完成后，在终端窗口中按 `Ctrl+C` 停止日志收集

6. **查看日志文件**：
   - 日志文件保存在 `/Users/yy/pro-test/ChineseChess/logs/` 目录下

## 7. Bug管理规范

### 7.1 Bug记录

- **记录位置**：所有Bug应记录在 `bug.md` 文件中
- **记录内容**：
  - Bug描述：详细描述Bug的表现
  - 复现步骤：如何复现Bug
  - 预期结果：应该出现的正确结果
  - 实际结果：实际出现的错误结果
  - 环境信息：设备型号、Android版本、应用版本
  - 日志信息：相关的日志内容

### 7.2 Bug分类

- **严重程度**：
  - 严重：导致应用崩溃或无法使用的Bug
  - 中等：影响功能但不导致崩溃的Bug
  - 轻微：影响用户体验但不影响功能的Bug

- **Bug类型**：
  - 功能Bug：功能无法正常工作
  - 界面Bug：界面显示异常
  - 性能Bug：应用响应缓慢或卡顿
  - 兼容性Bug：在特定设备或Android版本上出现的Bug

### 7.3 Bug修复流程

1. **Bug分析**：
   - 分析Bug原因和影响范围
   - 查看相关日志和代码

2. **Bug修复**：
   - 按照代码规范修复Bug
   - 确保修复不会引入新问题

3. **Bug验证**：
   - 测试修复效果
   - 确认Bug是否彻底解决

4. **文档更新**：
   - 在 `bug.md` 中标记Bug状态为已修复
   - 更新相关文档

### 7.4 Bug状态

- **待修复**：Bug已发现但未修复
- **修复中**：Bug正在修复中
- **已修复**：Bug已修复但未验证
- **已验证**：Bug已修复且验证通过
- **关闭**：Bug已彻底解决
